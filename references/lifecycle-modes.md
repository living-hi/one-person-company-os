# Lifecycle Modes

## Purpose

Select the right operating mode for the current stage of the company. Do not run validation, delivery, and growth with the same cadence or success criteria.

## Mode: Validate

### Goal

Reduce uncertainty about customer, problem, offer, and price.

### Main Questions

- Who has the problem?
- How painful and frequent is it?
- What outcome do they pay for?
- What wedge gets the first conversations or signups?

### Main Activities

- define ICP
- map pains and desired outcomes
- build the offer and pricing hypothesis
- run interviews, outreach, or landing page tests
- collect objections and evidence

### Exit Criteria

- a specific customer segment is chosen
- the offer is understandable
- demand signals exist
- the next MVP is justified

## Mode: Build

### Goal

Ship the smallest useful version of the product with enough quality to learn or sell.

### Main Questions

- What is the smallest release that solves the chosen problem?
- What scope is unnecessary right now?
- What architecture keeps delivery fast without creating avoidable risk?

### Main Activities

- write PRD
- define release scope
- design flows
- implement
- test
- prepare deployment

### Exit Criteria

- MVP or release candidate exists
- core flows are tested
- launch prerequisites are documented

## Mode: Launch

### Goal

Turn the shipped product into a coordinated release with clear positioning, demand capture, and support readiness.

### Main Questions

- Why should the target user care now?
- Which channel gets the fastest feedback loop?
- What onboarding or support gaps would kill activation?

### Main Activities

- define launch narrative
- prepare pages and content
- build lead capture and onboarding
- prepare support responses
- execute the launch checklist

### Exit Criteria

- launch assets are ready
- channel plan is defined
- support and measurement are ready

## Mode: Operate

### Goal

Run the company reliably and keep feedback loops tight.

### Main Questions

- Are users getting value repeatedly?
- What is breaking, slowing down, or creating support load?
- Which metrics actually predict health?

### Main Activities

- monitor reliability
- handle support
- review metrics
- summarize feedback
- reprioritize backlog

### Exit Criteria

- weekly operating rhythm exists
- incidents and support are handled repeatably
- backlog changes reflect evidence

## Mode: Grow

### Goal

Increase revenue, retention, and leverage without losing operating discipline.

### Main Questions

- Which channel or loop is most likely to compound?
- Where does the funnel leak?
- Which retention or pricing levers matter now?

### Main Activities

- run growth experiments
- improve funnel steps
- review retention
- strengthen sales pipeline
- automate repeated work carefully

### Exit Criteria

- a repeatable growth loop is emerging
- metrics improve through deliberate experiments
- leverage rises without uncontrolled complexity

## Selection Rule

Choose the mode based on the current bottleneck, not on founder mood.

Examples:
- many ideas but no evidence -> `Validate`
- clear demand but no usable product -> `Build`
- product exists but traction is weak because distribution is undefined -> `Launch`
- users exist but issues, churn, and support chaos dominate -> `Operate`
- retention is decent and demand exists but growth has plateaued -> `Grow`
