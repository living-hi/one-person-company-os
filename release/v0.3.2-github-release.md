# v0.3.2 - State Transparency and Runtime Recovery

This release makes `one-person-company-os` much more reliable in real use.

The core content and company-OS framing were already stronger in V2.
What was still missing was delivery trust:

- where exactly the workflow is right now
- whether key artifacts were really saved
- why some content was only in chat
- what to do when local Python/runtime setup is not compatible

This patch closes those gaps.

## What Changed

### 1. The workflow is now explicitly state-driven

Every real run now follows:

- `Step 1/5` mode selection
- `Step 2/5` preflight and persistence strategy check
- `Step 3/5` draft / proposal / current state load
- `Step 4/5` execution and persistence
- `Step 5/5` verification and reporting

This applies across:

- preflight
- company creation
- round start
- round update
- calibration
- checkpoint save
- stage transition

### 2. Save transparency is now first-class

The workflow now reports:

- whether content was saved
- where it was saved
- which files were written
- why content is still chat-only when it was not persisted

This is a meaningful product shift because the skill is now much less likely to feel like “a lot of output with unclear system state”.

### 3. Runtime reporting is much more precise

The runtime layer now distinguishes:

- `installed`
- `runnable`
- `python_supported`
- `workspace_created`
- `persisted`

That makes it clearer whether the environment is merely present versus truly usable.

### 4. Python recovery is now built in

New entry points:

- `scripts/preflight_check.py`
- `scripts/ensure_python_runtime.py`
- `scripts/checkpoint_save.py`

`scripts/ensure_python_runtime.py` now supports:

- discovering compatible local Python interpreters
- generating install plans for common platforms
- letting OpenClaw switch interpreters or recover the workflow instead of stopping at a version error

### 5. The repository front page is now sharper

The README has been rewritten to target founders actively building AI-native solo companies.

It now leads with:

- founder situation
- product promise
- first-run output
- state/save/runtime transparency
- Python recovery behavior

That makes the project much more compelling and much easier to understand at first glance.
